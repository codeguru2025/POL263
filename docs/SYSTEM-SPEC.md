# POL263 — System Specification & Architecture Overview

> **Purpose of this document.** A single, self-contained briefing for an external advisor/consultant.
> It describes what POL263 is, how it is built, how it is laid out, and where the architectural
> seams and risks are — enough to assess the system and recommend improvements without first
> reading the code.
>
> **Status:** Living document. Figures (table/route counts) reflect the codebase at the time of
> writing and are derived directly from source, not aspirational.

---

## 1. Executive Summary

POL263 is a **multi-tenant Policy Management System (SaaS)** built for insurance companies and
funeral-assurance societies, with a strong Zimbabwean/Southern-African market focus (PayNow mobile
money, USD/ZAR/ZIG multi-currency, funeral cover, burial societies).

A single TypeScript codebase serves **three client surfaces**:

| Surface | Audience | Auth |
|---|---|---|
| **Staff portal** | Insurer back-office (admins, managers, cashiers, claims, finance) | Google OAuth only |
| **Agent portal / app** | Field sales agents (web + native Android/iOS via Capacitor) | Google OAuth |
| **Client portal** | Policyholders (self-service) | Email/password or Google OAuth |

The same codebase compiles to web **and** native mobile (Capacitor), and supports
**white-labelling** so each tenant can run under its own brand.

**Scale of the system (current):**
- **81** database tables (Drizzle ORM / PostgreSQL)
- **~226** HTTP API endpoints (single Express registration file)
- **~273** data-access methods in the storage layer
- **27** staff pages + dedicated agent and client portals
- **54** discrete RBAC permissions across **9** seeded roles

### 3.3 Multi-tenancy model

Two layers of isolation, both present:

1. **Row-level scoping (default).** Every domain table carries `organization_id` (and often
   `branch_id`). The `requireTenantScope()` middleware enforces that the authenticated user
   belongs to the requested org on every call. This is the primary isolation mechanism.
2. **Database-level isolation (optional, per tenant).** A tenant may have its own dedicated
   database (`organizations.databaseUrl`). `tenant-db.ts` routes that tenant's queries to its
   isolated DB. A **control-plane** database holds the shared org/user registry.

A **Platform Owner** (super-account, set via `SUPERUSER_EMAIL`) sits *above* all tenants: it
implicitly holds every permission plus platform powers (`create:tenant`, `delete:tenant`,
`manage:whitelabel`) and can switch into any tenant ("control-plane mode"). Tenant **superusers**
have full access only *within* their own tenant.

### 3.4 Request lifecycle (typical mutation)

```
Client (TanStack Query, CSRF token injected)
  → Express route (routes.ts)
    → requireAuth → requireTenantScope → requirePermission("write:x")
      → storage.ts method (Drizzle, org-scoped query)
        → auditLog(action, entityType, id, before, after)   [JSONB delta]
        → outbox/job-queue enqueue (if async side-effects, e.g. notifications)
  → JSON response
```

## 4. Authentication & Authorisation (RBAC)

### 4.1 Authentication
- **Staff & agents:** Google OAuth only (no passwords) — `server/auth.ts`.
- **Clients:** email/password (Argon2) or Google OAuth — `server/client-auth.ts`, with security
  questions for recovery.
- Sessions are stored in PostgreSQL (`connect-pg-simple`); CSRF protection via `csurf`; the client
  injects the CSRF token on every mutating request.

### 4.2 RBAC model (database-driven)

```
users → user_roles → roles → role_permissions → permissions
                 ↘ user_permission_overrides (per-user grant/revoke)
```

- Roles and role→permission mappings live in the database (seeded from `constants.ts`).
- `user_roles` can be scoped to a **branch** (`branch_id` nullable → org-wide).
- `user_permission_overrides` allows granting *or explicitly revoking* a single permission for one
  user, layered on top of role permissions.
- Effective permissions are computed server-side (`getUserEffectivePermissions`) and also drive the
  **frontend navigation visibility** (menu items are filtered by permission).

### 4.3 Permission catalogue (54 permissions, by category)

| Category | Permissions |
|---|---|
| organization | read/write:organization, read/write:branch |
| identity | read/write/delete:user |
| rbac | read/write:role, manage:permissions |
| audit | read:audit_log |
| policy | read/write:policy, edit:premium, delete:policy |
| claims | read/write:claim, approve:claim |
| clients | read/write:client, view:own_clients, view:all_clients |
| product | read/write:product |
| settings | manage:settings |
| operations | read/write:funeral_ops |
| finance | read/write:finance, **approve:finance**, delete/edit:payment, delete/edit:receipt, backdate:payment, receipt:cash/mobile/transfer/group |
| fleet | read/write:fleet |
| commission | read/write:commission |
| payroll | read/write:payroll |
| reports | read/write:report |
| leads | read/write:lead |
| notifications | read/write:notification |
| approvals | manage:approvals |
| platform | create:tenant, delete:tenant |

### 4.4 Seeded roles (9)

| Role | Scope intent |
|---|---|
| **superuser** | Full tenant access (all permissions within the org) |
| **administrator** | Org admin: full read/write incl. finance approve, delete:policy/payment, RBAC mgmt |
| **manager** | Operational management: read-most, write claims/policies/clients, approvals, receipts |
| **executive** | Read-only across the org (reporting/oversight) |
| **cashier** | Receipting (cash/mobile/transfer/group) + read finance/policy/client |
| **agent** | Field sales: own clients, write policy/lead/client, mobile/transfer receipts, read commission |
| **claims_officer** | Claims lifecycle + funeral ops |
| **fleet_ops** | Fleet + funeral ops |
| **driver** | Read-only funeral cases + fleet they're dispatched with |

**Maker-checker** is enforced for high-risk actions: `approve:claim` and `approve:finance` are
required at runtime (separate from the create permission), so the person who raises a requisition
or claim cannot also approve it.

## 6. Functional Modules

API routes are organised into clearly-labelled domain sections in `server/routes.ts`. The main
functional modules:

| Module | Key capabilities |
|---|---|
| **Policy administration** | Create/edit policies, members, add-ons, status history, hard-delete (RBAC-gated), premium overrides |
| **Clients & dependents** | Client CRM, dependents/beneficiaries, dependent change requests, client documents (ID, proof of address) |
| **Products & price book** | Versioned products, benefit catalog/bundles, add-ons, age bands, cost sheets, price book |
| **Payments & receipting** | Manual payments, receipts, PayNow intents/polling, cash/mobile/transfer/group receipts, edit/delete (RBAC-gated, audited) |
| **Month-end close** | Batch receipt from bank file upload; premium collection run for overdue policies |
| **Group / employer schemes** | Group batch receipting, group PayNow, payment allocations, settlements |
| **Cash-up reconciliation** | Daily cash count vs receipts issued |
| **Requisitions** | Expenditure request → submit → approve/reject (maker-checker) → mark paid |
| **Expenditures & FX rates** | Operating expenses; USD-base FX rates for consolidated statements |
| **Claims** | Claim capture, adjudication, approval (maker-checker), documents, status history |
| **Funeral operations** | Funeral cases, tasks, quotations + items, service receipts (cash-service income) |
| **Fleet** | Vehicles, driver assignments, fuel logs, maintenance |
| **Commissions & platform revenue** | Commission plans, ledger entries, platform receivables / revenue share, settlements |
| **Payroll** | Employees, payroll runs, payslips |
| **Leads / pipeline** | Sales pipeline, agent referral links, public registration from referral |
| **Notifications** | SMS templates + logs |
| **Approvals** | Generic maker-checker approval queue |
| **Reports** | CSV/dynamic reports, policy/finance/employee report sets, consolidated financial statements |
| **Diagnostics & audit** | System issue reports, full audit trail viewer, admin diagnostics |
| **Tenant / platform admin** | Org settings, branches, users, roles, tenant switching, app release management, terms & conditions |

### Cross-cutting concerns

- **Audit logging.** Every mutation calls `auditLog(action, entityType, entityId, before, after)`;
  before/after stored as JSONB delta with a request id for tracing. A staff audit viewer exists.
- **Idempotent payments.** PayNow creation uses an idempotency key; status is **polled**
  (PayNow has no webhooks) via `pollPaynowStatus()`. On confirmation, `applyPaymentToPolicy()`
  transitions policy status. `PAYNOW_INTEGRATION_KEY` is never exposed to the client.
- **Transactional outbox + job queue.** Side-effects (notifications, async work) are enqueued via
  an outbox pattern so they survive failures and are processed out-of-band.
- **Document generation & storage.** PDFs (receipts, policy docs, funeral docs) generated with
  PDFKit and stored in S3.
- **Rate limiting.** Redis-backed when `REDIS_URL` is set; otherwise per-process in-memory.

## 8. Operations & Deployment

- **Build:** `script/build.ts` (esbuild server bundle → `dist/index.cjs`, Vite client build).
- **Run:** `npm run start` (prod) / `start:with-migrate` (migrate then start).
- **Migrations:** Drizzle Kit. Multiple configs exist for control-plane, tenant, and per-tenant
  databases (`drizzle.*.config.ts`). `db:migrate:status` reports drift vs disk.
- **Seeding:** `db:seed` loads RBAC roles/permissions and the platform owner.
- **Environments documented:** DigitalOcean App Platform (primary), Interserver VPS, Netlify
  (client), plus a Supabase→DigitalOcean DB migration runbook.
- **Required env:** `DATABASE_URL`, `SESSION_SECRET`, `SUPERUSER_EMAIL` (prod).
  Optional: `GOOGLE_CLIENT_ID/SECRET`, `PAYNOW_INTEGRATION_ID/KEY`, `REDIS_URL`,
  `AWS_S3_BUCKET/ACCESS_KEY/SECRET`.
- **CI/quality gates:** Husky pre-commit runs `build` + `test`; lockfile integrity checks.

## 9. Observations & Areas for the Advisor to Review

These are candid, code-derived observations — starting points for an improvement engagement, not
defects per se.

1. **Monolithic route & storage files.** `routes.ts` (~241 KB, 226 endpoints) and `storage.ts`
   (~163 KB, 273 methods) are single files. They are well-sectioned but are a maintainability and
   merge-conflict risk, and make ownership/testing boundaries hard. *Candidate for modular split
   by domain (policies, finance, claims, …).*

2. **Two tenancy models coexisting.** Row-level `organization_id` scoping is the default, but
   per-tenant isolated databases are also supported. The operational and backup/restore complexity
   of running both should be assessed (consistency of migrations across tenant DBs, the control
   plane as a single point of coordination).

3. **PayNow polling, not webhooks.** Payment confirmation relies on polling. Worth reviewing the
   polling cadence, reconciliation guarantees, and what happens to in-flight intents on restart.

4. **Async via custom outbox + job queue.** There is a hand-rolled outbox/job-queue rather than a
   managed broker. Review durability, retry/back-off, dead-lettering, and observability.

5. **Auth surface.** Staff/agents are Google-OAuth-only (good), but clients use email/password +
   OAuth + security questions — review password reset, lockout, and brute-force protections beyond
   rate limiting.

6. **Permission breadth.** 54 permissions × 9 roles + per-user overrides is flexible but can drift.
   Worth a review of least-privilege defaults and an audit of override usage.

7. **Reporting load.** Reports include CSV exports and consolidated multi-currency statements built
   in-process (`financial-statements.ts`). Assess query cost / read-replica or pre-aggregation needs
   at scale.

8. **Test coverage vs surface area.** Vitest unit/e2e + TestSprite exist; given 226 endpoints, the
   ratio of automated coverage to surface area is worth measuring (RBAC and money paths especially).

9. **Single mega-PRD + many ad-hoc docs.** `docs/` contains 30+ documents including overlapping
   reports (SCALABILITY, OVERHAUL, REFACTOR-PROGRESS, DEPLOY-ANALYSIS). Consolidating into a
   maintained source-of-truth set would help onboarding.
