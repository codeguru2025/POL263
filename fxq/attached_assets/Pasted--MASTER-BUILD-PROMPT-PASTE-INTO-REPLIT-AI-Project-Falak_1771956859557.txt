=====================================================================
A) NON-NEGOTIABLE ENGINEERING RULES (PROGRESSIVE, SAFE, DEBUGGABLE)
=====================================================================

1) Progressive delivery (no breaking changes)
- Implement in small phases with feature flags.
- Never break existing endpoints or DB schemas without migrations and tests.
- Every phase must ship: migrations, unit tests, and at least one e2e test for the new critical path.
- Use semantic versioning in the repo tags/releases.
- Keep old behavior behind flags until new behavior is verified.

2) Configuration-first (no hard-coded business rules)
- Products, benefits, pricing, add-ons, eligibility, waiting periods, grace/lapse/reinstatement, commissions, age bands, claim requirements, exclusions, notification templates, price book items, costing rules must be DB-configurable via Admin Builders with versioning + effective dates + rollback.
- Code is an engine; business is configuration.

3) Financial integrity (immutable ledger)
- NEVER edit or delete financial transactions.
- Corrections happen via VOID/REVERSAL entries (negative entries).
- Every receipt has immutable identity; voiding requires reason, approver (maker-checker), and audit.
- Daily cashups can be locked; unlocking is logged and restricted.

4) Security-first / Zero-trust
- All authorization is server-side. Never trust front-end.
- Enforce RBAC + row-level access checks on EVERY API route.
- Prevent IDOR: use UUIDs, signed references where needed.
- Rate limit auth, resets, webhooks.
- Strict CSP, secure cookies, SameSite, CSRF protection, input validation.
- Hardened against privilege escalation and lateral movement between tenants/roles.

5) Debuggability
- Structured logging with request_id on every request; include user_id, role, tenant_id, policy_id where relevant.
- Central error handler, safe user-facing messages, detailed server logs.
- Admin diagnostics screens: failed webhooks, queue backlog, notification failures, reconciliation mismatches.
- Audit log viewer: searchable, filterable, before/after diff.

G) POLICY LIFECYCLE, BILLING, GRACE/LAPSE, REINSTATEMENT (LOCKED + MATH)
=====================================================================

G1) Policy status state machine (enforced)
Allowed transitions:
- Draft → Pending → Active
- Active → Grace → Active
- Grace → Lapsed
- Lapsed → ReinstatementPending → Active (after approval/auto-conditions)
- Active/Grace/Lapsed → Cancelled (if allowed; configurable)
Illegal transitions must be rejected.

G2) 30-day billing cycle with installment payments (mathematical definition)
Variables:
- Pm = monthly premium amount for the policy’s currency/product version.
- CycleStart = the start date of the current billing cycle.
- CycleEnd = CycleStart + 30 days.
- GraceEnd = CycleEnd + 30 days.
- Each PaymentTransaction has:
  - received_at (timestamp recorded/cleared; used for financial reporting & cashups)
  - posted_date (date it hits books; default = received_at date)
  - value_date (effective date used for policy compliance allocation; can be backdated with approval)
- ΣCyclePaid = sum(amount allocated to this policy with value_date ∈ [CycleStart, CycleEnd] and status=cleared)
- ΣPaidByGrace = sum(amount allocated to this policy with value_date ∈ [CycleStart, GraceEnd] and status=cleared)

Rules:
1) Compliant by Cycle End if ΣCyclePaid ≥ Pm.
2) If ΣCyclePaid < Pm at CycleEnd → status becomes GRACE immediately.
3) If ΣPaidByGrace ≥ Pm at any time during grace → status returns to ACTIVE (subject to reinstatement rule if it had lapsed).
4) If ΣPaidByGrace < Pm at GraceEnd → status becomes LAPSED.

Carry forward:
- If payment exceeds remaining amount needed for current cycle, allocate remainder to next cycle as Advance Credit (configurable behavior; default: carry forward).

G3) Lapse behavior (LOCKED)
On LAPSED:
- policyholder + dependents are frozen (no benefits; no edits except reinstatement workflow)
- commissions clawback logic applies (below)

G4) Automatic revival on arrears (LOCKED)
- If arrears are fully paid (ΣPaidByGrace reaches required amounts for lapsed cycles), system automatically triggers reinstatement workflow and restores policy to ACTIVE (with audit and any required approvals).
- Reinstatement triggers NEW waiting period (LOCKED TRUE by default; configurable).

G5) Backdated payments WITHOUT distorting cashups/reports (LOCKED)
- Backdating changes value_date only.
- Financial reports and daily cashups are computed using received_at/posted_date.
- Backdating requires: permission + reason + maker-checker approval + audit.
- Max backdate window configurable.
