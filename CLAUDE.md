## Project Overview

POL263 is a multi-tenant Policy Management System (SaaS) for insurance companies. It supports web, Android, and iOS from a single codebase via Capacitor. The stack is TypeScript throughout: React 19 + Vite (frontend), Express 5 (backend), Drizzle ORM on PostgreSQL.

## Commands

```bash
# Development
npm run dev                  # Start backend + frontend dev server
npm run dev:client           # Frontend only (Vite, port 5000)

# Build
npm run build                # Full build (client + server)
npm run check                # TypeScript type check

# Testing
npm run test                 # Run all tests once (Vitest)
npm run test:watch           # Watch mode
npm run test -- rbac.test.ts # Run a single test file

# Database
npm run db:push              # Apply schema changes (Drizzle)
npm run db:migrate           # Run SQL migrations (DATABASE_URL; also DATABASE_URL_TENANT if set and distinct)
npm run db:migrate:status    # List pending/applied migrations vs disk (optional second DB)
npm run db:seed              # Seed initial data
npm run db:reset             # Reset DB (keeps platform owner)

# Mobile (Capacitor)
npm run cap:sync             # Sync web build to native projects
npm run cap:android          # Open Android Studio
npm run cap:ios              # Open Xcode

# Production
npm run start:with-migrate   # Run migrations then start server
```

Tests live in `tests/unit/` and `tests/e2e/`. Vitest config is at `vitest.config.ts`.

## Architecture

### Monorepo Layout

- `client/` — React 19 frontend (Vite)
- `server/` — Express 5 backend
- `shared/` — Code shared between client and server
  - `schema.ts` — All Drizzle ORM table definitions (~2000 lines, source of truth)
  - `validation.ts` — Shared Zod validation (national ID, currency)
- `migrations/` — SQL files generated by Drizzle
- `tests/` — Unit and E2E tests
- `docs/` — Architecture docs, deployment guides, ERD

### Backend Key Files

| File | Purpose |
|---|---|
| `server/routes.ts` | All API route handlers (~241KB, central registration point) |
| `server/storage.ts` | Database access layer (~163KB, all Drizzle queries) |
| `server/auth.ts` | Staff authentication (Google OAuth via Passport) |
| `server/client-auth.ts` | Client authentication (email/password + Google OAuth) |
| `server/payment-service.ts` | PayNow payment initiation and polling |
| `server/route-helpers.ts` | Shared middleware: audit logging, permission checks, calculations |
| `server/tenant-db.ts` | Multi-tenancy: routes queries to per-org database if provisioned |

### Frontend Key Files

- `client/src/App.tsx` — Route configuration with lazy-loaded page components
- `client/src/lib/queryClient.ts` — TanStack Query setup with CSRF token injection
- `client/src/hooks/` — `useAuth()`, `useBranding()`, `useMobile()`
- `client/src/pages/staff/` — Staff dashboard pages
- `client/src/pages/agent/` — Agent portal pages
- `client/src/pages/client/` — Client portal pages

### Multi-Tenancy

Every database query is scoped by `organization_id`. Optionally, tenants can have an isolated database (`organizations.databaseUrl`). The `requireTenantScope()` middleware enforces that the authenticated user belongs to the requested org. Roles and permissions can be further scoped to `branch_id`.

### Authentication & RBAC

- **Staff**: Google OAuth only (no passwords) — configured in `server/auth.ts`
- **Clients**: Email/password or Google OAuth — `server/client-auth.ts`
- Sessions stored in PostgreSQL via `connect-pg-simple`
- RBAC is database-driven: `users → user_roles → roles → role_permissions → permissions`
- Per-user permission overrides supported via `user_permission_overrides`
- Route guards: `requireAuth()`, `requirePermission()`, `requireAnyPermission()` in `route-helpers.ts`

### Payments (PayNow)

PayNow is a Zimbabwean payment gateway. Key behaviors:
- Payment creation is **idempotent** (idempotency key prevents duplicates)
- Status updates are **polled** — PayNow does not support webhooks; `pollPaynowStatus()` handles this
- On confirmation, `applyPaymentToPolicy()` transitions policy status
- `PAYNOW_INTEGRATION_KEY` must never be exposed to the client

### Audit Logging

Every data mutation calls `auditLog(action, entityType, entityId, before, after)`. Before/after states are stored as JSONB. The staff UI has an audit viewer. Include audit logging in any new mutation routes.

### Data Access Pattern

All database operations go through `server/storage.ts`. Routes call storage methods rather than querying the database directly. Zod schemas for inserts are auto-generated via `drizzle-zod` from `shared/schema.ts`.

### Product Pricing

Products have versioned configurations (`products → product_versions`). Premium = base + age band adjustment + add-on prices. Supports multiple currencies (USD, ZAR, ZIG) and payment schedules (monthly, weekly, biweekly, yearly).

## Environment Variables

See `.env.example`. Required variables:

```bash
DATABASE_URL=postgresql://...
SESSION_SECRET=...           # Generate with: npm run generate-secret
```

Key optional variables:
```bash
GOOGLE_CLIENT_ID / GOOGLE_CLIENT_SECRET   # Staff login
PAYNOW_INTEGRATION_ID / PAYNOW_INTEGRATION_KEY  # Payments (server-side only)
REDIS_URL                    # Distributed rate limiting (falls back to in-memory)
AWS_S3_BUCKET / AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY  # Document storage
```

## Database

Schema is defined in `shared/schema.ts` using Drizzle ORM. To make schema changes:
1. Edit `shared/schema.ts`
2. Run `npm run db:push` (development) or generate a migration with `npm run db:migrate`

Session table is managed by `connect-pg-simple`. Rate limiting optionally uses Redis; without it, limits are per-process.
